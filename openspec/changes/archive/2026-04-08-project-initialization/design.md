## Context

当前仓库尚未形成可开发的前端应用工程，缺少统一的应用骨架、工程规范与质量门禁。基于 `proposal.md`，本次设计需要在“可快速起步”和“后续可扩展”之间取得平衡，并为团队协作提供一致的开发入口。

约束条件：

- 以 React + TypeScript 为主技术栈。
- 优先保证初始化阶段的稳定性与可维护性，不引入过重架构。
- 初始化结果需可被 CI 直接消费（脚本标准化、检查流程明确）。

相关方：

- 前端开发：需要统一目录与脚手架。
- QA/CI：需要稳定、可重复的检查与构建命令。
- 后续业务模块开发者：需要清晰扩展点与配置约定。

## Goals / Non-Goals

**Goals:**

- 提供最小可运行的前端应用骨架（入口、页面结构、基础路由占位）。
- 建立统一工程质量基线（lint、format、type-check、构建脚本）。
- 定义环境配置与文档约定，降低新成员接入成本。
- 为后续业务模块预留扩展结构（模块目录与通用布局）。

**Non-Goals:**

- 不实现具体业务功能与复杂交互。
- 不在本阶段引入微前端、状态管理中台等重型架构。
- 不绑定特定后端接口，仅预留 API 对接层结构。

## Decisions

1. 采用“分层但轻量”的目录结构
   - Decision: 使用 `app/pages/components/services/config` 等基础分层。
   - Rationale: 兼顾可读性和扩展性，避免过度设计。
   - Alternative considered: 按业务域完全拆分（feature-first）在初始化阶段会增加认知成本，暂不采用。

2. 工程质量门禁前置到本地与 CI
   - Decision: 统一提供 `lint`、`format`、`type-check`、`build` 脚本，并作为提交前/流水线基础检查。
   - Rationale: 早期统一规范可显著减少返工与风格分歧。
   - Alternative considered: 仅依赖 CI 检查会导致反馈滞后，开发体验较差。

3. 环境配置采用模板化管理
   - Decision: 提供 `.env.example` 与配置读取约定，避免硬编码。
   - Rationale: 便于多环境切换与安全治理。
   - Alternative considered: 直接在代码写死配置会增加泄漏风险与维护成本。

4. UI 能力采用“可替换”接入策略
   - Decision: 在初始化阶段先定义通用页面框架与基础组件边界，避免深度耦合。
   - Rationale: 后续可平滑接入团队组件库并逐步替换。
   - Alternative considered: 一次性绑定复杂 UI 体系会拖慢初始化交付。

## Risks / Trade-offs

- [初始化结构与后续业务不完全匹配] → 通过明确目录约定与渐进式重构规则降低调整成本。
- [规则过严影响初期迭代速度] → 仅启用核心门禁（lint/type-check/build），复杂规则后续分阶段增强。
- [环境变量约定执行不一致] → 在 README 提供示例与检查步骤，CI 增加基础校验。
- [预留层次过多导致样板代码增加] → 控制为最小骨架，仅保留必需目录与入口文件。

## Migration Plan

1. 建立应用骨架与基础目录。
2. 配置工程脚本与质量检查工具。
3. 增加环境变量模板与配置读取层。
4. 补充 README 的启动、构建、检查说明。
5. 在 CI 中接入基础检查命令。

回滚策略：

- 若初始化方案影响现有流程，可按提交粒度回滚（先回滚 CI 检查，再回滚工程配置，最后回滚目录结构变更）。

## Open Questions

- 是否在初始化阶段直接确定路由方案（如 React Router）还是先保留单页入口占位？
- 是否需要在初始化阶段同时引入单元测试框架（Vitest/Jest）作为强制门禁？
- 组件库接入优先级：初始化即接入还是在首个业务模块开发前接入？
